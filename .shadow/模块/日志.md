# 模块：日志

> 日志就是"记下来发生了什么"。**没日志的爬虫等于盲飞**——挂了不知道哪里挂的，慢了不知道慢在哪一步。

## 当前状态

[src/util/logger.ts](../../src/util/logger.ts) 提供一个**唯一**的全局 `logger`：

- 底层用 [pino](https://getpino.io)，结构化 JSON 输出。
- 同时写两个 sink（`pino.multistream`）：
  - **stdout** ——开发时看
  - **文件** `./.logs/app/app-YYYY-MM-DD.log` ——日切，落盘
- 内置脱敏：`addRedactedSecret(secret)` 把密码塞进去之后，任何 pino bindings 里的字符串字段都会自动替换成 `***`。

测试 / CI 环境（`NODE_ENV=test` 或 `VITEST=true` 或 `LOG_FILE_DISABLED=true`）**自动跳过文件 sink**，只写 stdout，避免污染工作区。

## 设计原则（user feedback 推动的简化版）

之前一度提议过分两个 logger——一个 pino 一个 "OpsLogger" 给非开发者看人话。**砍掉了**——多一个名字就多一份心智负担，所有调用都走同一个 `logger`，让 pino 同时干两件事：

- 给开发者看：调高 LOG_LEVEL 看 `debug`，pino-pretty 管漂亮打印；
- 给运维 / 非开发者看：直接 grep `./.logs/app/*.log`，按 `"level":40` 这种字段筛 warn/error。

## 调用约定

脚本里典型用法：

```typescript
import { logger } from '../../src/util/logger';

logger.info({ siteKey, count: 23 }, 'init-sections: 完成');
logger.warn({ boardKey, err: '超时' }, '版面抓取超时，跳过');
logger.error({ url, err: String(e) }, '帖子抓取失败');
```

**两参数模式**：第一个是"绑定字段"（结构化 JSON，便于 grep），第二个是给人看的中文短句。两者都进日志。

**`console.error` 何时仍可用**：

1. **CLI 用法提示**（`Usage: tsx scripts/xxx.ts <arg>`）：脚本第一步参数校验失败时打到 stderr，符合 Unix 习惯。
2. **顶层 `main().catch`**：进程即将 `exit(1)`，pino 异步写文件可能来不及 flush，用 stderr 兜底。

中间流程其它一切错误都走 `logger.error`。

## 日志级别约定

| 级别 | 何时用 | 示例 |
|---|---|---|
| `debug` | 详细排查（默认不开） | "翻到第 3 页"、"selector 命中 12 行" |
| `info` | 正常进展 | "init: 入库 23 个讨论区"、"login successful"、"版面 X 抓 45 帖" |
| `warn` | 不致命但要留意 | "置顶帖抓取失败，跳过"、"selector 0 行"、"进度文件写入失败" |
| `error` | 任务失败 | "重试 3 次仍失败"、"数据库写入失败" |

## 文件结构

```
./.logs/
  app/
    app-2026-05-11.log         ← 当天所有 info+ 日志
    app-2026-05-12.log
    ...
```

按 UTC 日期切。日切实现是"启动时按当天日期开文件"——如果一个进程跨午夜，**当前实现不会自动切换文件**。本项目的脚本是短跑作业（最长几十分钟的 `init:pinned`），不会跨日。如果以后跑长驻服务，再接 [pino-roll](https://github.com/mcollina/pino-roll) 之类的自动切割。

## 脱敏

实现在 [logger.ts:9-21](../../src/util/logger.ts)：

```typescript
const secrets = new Set<string>();
export function addRedactedSecret(secret: string) { secrets.add(secret); }
export function redactString(input: string) {
  for (const s of secrets) input = input.replace(new RegExp(escapeRegex(s), 'g'), '***');
  return input;
}
```

pino formatter 跑在每条日志的"合并对象"上：

```typescript
formatters: {
  log(obj) {
    // 把所有 string 字段过一遍 redactString
  },
},
```

⚠️ **已知局限**：pino 的 `msg` 字段是后加入的、不经过这个 formatter，所以**写在 message 字符串里的密码不会被脱敏**。现有所有调用方都把变量放在 bindings（`logger.info({ password }, '...')`），不写进 message——实践中没有泄漏。如要彻底堵这个口，下一步可以接 pino 的 `redact` 选项 + 自定义 message 序列化器。

`AuthManager` 一拿到密码立即调 `addRedactedSecret(creds.password)`，所以之后所有路径都受保护（参见 [浏览器与会话.md](浏览器与会话.md)）。

## `fetch_log` 表（不是运行日志）

[src/repository/fetch-log.ts](../../src/repository/fetch-log.ts) 维护一张数据库表，记**外部 API 调用审计**——MCP 项目以后调本项目接口时，每次调用一行：

| 字段 | 含义 |
|---|---|
| `site_key` | 调的哪家 site |
| `tool` | 调的哪个工具方法（`forum_get_thread` / `forum_list_threads` / ...） |
| `args` | 调用参数（JSON） |
| `status` | `ok` / `error` / `rate_limited` |
| `error_code` | 失败时的 `BaseAppError.code` |
| `duration_ms` | 耗时 |
| `created_at` | 时间 |

**这是调用审计，不是运行日志**，两者用途完全不同。本模块不动它，详见 [数据库访问层.md](数据库访问层.md)。

## 配置项

| env | 默认 | 含义 |
|---|---|---|
| `LOG_LEVEL` | `info` | pino 最低级别（`debug`/`info`/`warn`/`error`） |
| `LOG_DIR` | `./.logs` | 日志根目录 |
| `LOG_FILE_DISABLED` | （未设） | 设为 `true` 跳过文件 sink |
| `NODE_ENV` / `VITEST` | （未设） | 这两个为 `test` / `true` 时也自动跳过文件 sink |
