# 模块：日志

> 日志就是"记下来发生了什么"。**没日志的爬虫等于盲飞**——挂了不知道哪里挂的，慢了不知道慢在哪一步。

## 当前状态 vs 目标状态

**当前**（[src/util/logger.ts](../../src/util/logger.ts)）：

- 用 [pino](https://getpino.io) 输出**结构化 JSON 到 stdout**（终端）。
- 有 `addRedactedSecret(s)` 让密码不写进日志。
- **不落盘**。终端关了就没了。

**目标**（你明确要求"再加一个日志以及保存"）：

- 仍然 stdout 输出（开发时方便看）。
- 增加**文件输出**：按日期切割，保留 N 天，每次"爬取/初始化操作"都写一行。
- 区分 **info / warn / error** 级别——按级别分文件，或同一文件按字段过滤。
- 关键事件可单独写"操作流水"日志（人类可读，给非开发者看）。

## 设计

### 文件结构（待落地）

```
./.logs/
  app/                       ← 通用应用日志
    app-2026-05-11.log
    app-2026-05-12.log
    ...
  ops/                       ← "操作流水"——给非开发者看的人类可读日志
    init-2026-05-11_143022.log     ← 一次完整 init 运行
    crawl-board-2026-05-11_150103.log
```

### 日志级别约定

| 级别 | 用途 | 示例 |
|---|---|---|
| `debug` | 详细调试（默认不开） | "翻到第 3 页"、"DOM 选择器命中 12 行" |
| `info` | 正常进展 | "init: persisted 23 sections"、"login successful"、"crawled board X, 45 threads" |
| `warn` | 不致命但要留意 | "pinned thread fetch failed; skipping"、"selector matched 0 rows" |
| `error` | 任务失败 | "crawl-board failed after 3 retries"、"DB write error" |

### 通用日志（app/）：pino 多 sink

把 pino 接到多个目的地：

```typescript
// 伪代码
const streams = [
  { level: 'debug', stream: process.stdout },                       // 终端
  { level: 'info',  stream: pino.destination('./.logs/app/app-' + today + '.log') },
];
const logger = pino({ level: 'debug', /* redaction */ }, pino.multistream(streams));
```

- 终端看 `debug` 以上（方便开发）。
- 文件只记 `info` 以上（生产场景）。
- 按日期切割：每天午夜（CST）切到新文件。简单做法是启动时按当天日期开文件；进程长跑时用 [pino-roll](https://github.com/mcollina/pino-roll) 之类的工具。

### 操作流水（ops/）：人类可读

pino 的 JSON 对开发者很方便，但对非开发者**不友好**。所以**每次执行一个"工作流程"**（init / crawl-board / crawl-section）时，额外开一个**人类可读**日志：

```
[2026-05-11 14:30:22] === init 开始 ===
[2026-05-11 14:30:22] 步骤 1/3: 抓讨论区清单
[2026-05-11 14:30:25] ✓ 已记录 23 个讨论区
[2026-05-11 14:30:25] 步骤 2/3: 抓版面结构
[2026-05-11 14:30:25]   - 本站十大 ... 12 个版面
[2026-05-11 14:30:42]   - 校园生活 ... 8 个版面
[2026-05-11 14:31:01]   - 社团 ... 31 个版面 (含 5 个子讨论区)
...
[2026-05-11 14:35:18] 步骤 3/3: 抓置顶帖
[2026-05-11 14:35:18]   开始并发抓取 96 个版面 (并发=16)
[2026-05-11 14:42:55]   完成 92 个，失败 4 个，进入第 2 轮重试...
[2026-05-11 14:43:30]   完成 4 个，0 失败
[2026-05-11 14:43:30] === init 结束（总耗时 13 分 8 秒）===
```

实现思路：

1. 在 `scripts/init/*.ts` 和 `scripts/crawl/*.ts` 入口处创建一个 `OpsLogger`。
2. `OpsLogger` 是个简单 wrapper：写人类可读的行 + 同时把同一信息以结构化形式 `logger.info(...)` 复制到 app 日志。
3. 命名：`<command>-<timestamp>.log`。

### 已有的"敏感字段脱敏"

已经实现 ([logger.ts:14-21](../../src/util/logger.ts))：

```typescript
const secrets = new Set<string>();

export function addRedactedSecret(secret: string): void {
  if (secret && secret.trim()) secrets.add(secret);
}

// pino formatter 里逐字段替换：
// 任何字符串值，包含 secret 就替换为 '***'
```

- AuthManager 拿到密码后**立即**调 `addRedactedSecret(creds.password)`。
- 这样后续任何带密码的对象进日志都会自动脱敏。
- 加密凭据 key、storageState JSON 不进日志（本来就不会传给 logger）。

### `fetch_log` 表（不在本模块范围）

[src/repository/fetch-log.ts](../../src/repository/fetch-log.ts) 维护一张数据库表，记**外部 API 调用审计**——MCP 项目以后调本项目接口时，每次调用一行：

| 字段 | 含义 |
|---|---|
| `site_key` | 调的哪家 site |
| `tool` | 调的哪个工具方法（`forum_get_thread` / `forum_list_threads` / ...） |
| `args` | 调用参数（JSON） |
| `status` | `ok` / `error` / `rate_limited` |
| `error_code` | 失败时的 `BaseAppError.code` |
| `duration_ms` | 耗时 |
| `created_at` | 时间 |

**这个不是运行日志**，是**调用审计**，用途完全不同。本模块不动它，详见 [数据库访问层.md](数据库访问层.md)。
