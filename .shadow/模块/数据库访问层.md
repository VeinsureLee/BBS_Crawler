# 模块：数据库访问层

> 这一层是"数据进出数据库的唯一出入口"。业务代码（爬虫、服务、脚本）想读/写 DB，都走 `src/repository/` 下的函数，不会直接拼 SQL。**这样的好处**：SQL 错了集中改一个地方；表结构改了，业务无感。

## 整体形态

```
src/repository/
  db.ts                  ← initDb / getStructureDb / getForumDb 单例池 + schema 常量
  sites.ts               ← sites 表 CRUD（structure.db）
  sections.ts            ← nodes 表的 forum + sub_forum 部分（structure.db）
  boards.ts              ← nodes 表的 board 部分（structure.db）+ resolveBoardRoute 路由
  threads.ts             ← 帖子（forum db）。upsert 返回 { threadId, forumDb }
  posts.ts               ← 楼层（forum db），接受 forumDb 参数
  board-crawl-state.ts   ← 每版面爬取进度（forum db，按 board_node_id 索引）
  boards-lookup.ts       ← 按 name/id 查 board 节点（structure.db）
  fetch-log.ts           ← 调用审计（structure.db，全局）
```

> Phase 3 已落地分层存储。整体结构见 [数据库.md](../数据库.md)。

## `db.ts`：连接层

提供一个最小通用 `Db` 接口：

```typescript
interface Db {
  query<T>(sql: string, params?: unknown[]): Promise<QueryResult<T>>;
  exec(sql: string): Promise<void>;                         // 多语句 DDL/DML
  transaction<R>(fn: (tx) => Promise<R>): Promise<R>;
  close(): Promise<void>;
}
```

特点：

- 实现用 [better-sqlite3](https://github.com/WiseLibs/better-sqlite3)（同步驱动，被包了一层 Promise）。**性能比 node-sqlite3 快 N 倍**，且支持 WAL。
- 启动时自动开 `journal_mode = WAL`（写不阻塞读）+ `foreign_keys = ON`。
- **占位符规约**：用户写 `$1, $2, ...`（postgres 风格），驱动里转成 `:1, :2, ...`（better-sqlite3 的命名参数）。这样如果以后回 postgres，业务代码一行不改。
- `transaction` 用 `BEGIN DEFERRED / COMMIT / ROLLBACK` 显式包裹。`exec` 不支持参数，给 schema 迁移用。

**单例**：`initDbs({ dataDir })` 初始化双库；`getStructureDb()` / `getContentDb()` 全局取实例。第一次没 init 调 get 会抛 `DatabaseError`。

## 每个 repository 文件的统一形态

```typescript
// 1. 类型：输入/输出的形状
export interface UpsertXxxInput { ... }
export interface UpsertXxxResult { ... }
export interface XxxRow { ... }

// 2. 查询函数（读）
export async function listXxx(...): Promise<XxxRow[]> {
  try {
    const r = await getXxxDb().query<...>(`SELECT ...`, [...]);
    return r.rows.map(rowToObject);
  } catch (e) {
    throw new DatabaseError('listXxx failed', e);
  }
}

// 3. 写入函数（先 SELECT 看是否存在 → UPDATE 或 INSERT）
export async function upsertXxx(input: ...): Promise<...> {
  try { ... } catch (e) { throw new DatabaseError('upsertXxx failed', e); }
}
```

约定：

- **所有错误用 `DatabaseError` 包一层**。上层只看 `e.code === 'DATABASE'` 就知道是 DB 问题，原始错误挂在 `e.cause`。
- **不直接 throw 一个字符串或 raw `Error`**——失去类型信息，logger 也会漏掉关键上下文。
- **返回数据用 driver-agnostic 形状**（自己定 `XxxRow`，不暴露 better-sqlite3 的具体行类型）。

## upsert 的写法

每个 entity 的 upsert 都长这样：

```typescript
// 1. SELECT 看是否存在
const exists = await db.query<{ id: number }>(
  `SELECT id FROM xxx WHERE site_key = $1 AND key = $2`, [siteKey, key],
);

if (exists.rows.length > 0) {
  // 2a. 存在 → UPDATE，关键字段用 COALESCE($n, col) 保留旧值
  await db.query(`UPDATE xxx SET name = $1, ... WHERE id = $2`, [...]);
  return { id: exists.rows[0]!.id };
} else {
  // 2b. 不存在 → INSERT
  await db.query(`INSERT INTO xxx (...) VALUES ($1, ...)`, [...]);
  const r = await db.query<{ id: number }>(`SELECT last_insert_rowid() as id`);
  return { id: r.rows[0]!.id };
}
```

**为什么不用 `INSERT ... ON CONFLICT ... DO UPDATE`**：

- SQLite 支持，但需要表上有 UNIQUE 约束才好用。
- 项目早期常常微调 schema，"SELECT-then-write"灵活，调试也直观。
- 性能差距在本项目体量下无感。

**`COALESCE($n, col)` 的作用**：如果传入 `null`，就保留 DB 里的旧值。例如更新置顶帖时不想把 `view_count` 抹掉，传 `null` 即可。

## 事务

只在"一次操作要写多行且必须原子"时用，例如 `upsertPosts(threadId, posts)`：

```typescript
await db.transaction(async (tx) => {
  for (const p of posts) {
    if (existsInTx) await tx.query(`UPDATE posts ...`);
    else            await tx.query(`INSERT INTO posts ...`);
  }
});
```

中途抛错 → 自动 ROLLBACK。

## 路由：跨 db 引用怎么找到对方

`threads.board_node_id` 指向 `structure.db.nodes.id` ——SQLite 不能跨文件强制 FK，靠**约定** + 路由函数：

- `resolveBoardRoute(siteKey, boardKey)` → `{ boardNodeId, forumDbFile }`：根据 boardKey 在 nodes 表里查到节点 id，再用递归 CTE 沿 `parent_id` 走到 `type='forum'` 那一行，读它的 `db_file`。
- `findForumDbFileForBoard(boardNodeId)` → `string | null`：直接拿 nodeId 也可以走同样的 CTE。

`upsertThread` 内部已自动调用 `resolveBoardRoute` 并返回 `forumDb`，调用方紧接着可以 `upsertPosts(forumDb, threadId, posts)` 不必再次解析。

## `fetch_log` 不是运行日志

容易混淆：

- **`fetch_log`**：每次有外部调用进 `CrawlerService` 时记一行（tool 名 / 参数 / status / duration / errorCode）。给"调用方"看的——例如 MCP 项目想查"agent 调了我哪些 API、几次失败"。Phase 3 之后从 content.db 搬到了 structure.db。
- **运行日志**（pino，写文件）：项目自己看的——"今天爬到一半磁盘满了" / "登录失败 3 次切换密码"。详见 [日志.md](日志.md)。
