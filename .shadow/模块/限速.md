# 模块：限速

> 论坛是别人的服务器，**疯狂访问会被封 IP / 触发风控 / 把人家服务器搞慢**。所以爬虫每次发请求前，要"先等一会"。这块就是管这个的。

## 一句话原理

**两个请求之间至少隔 `minIntervalMs` 毫秒，外加一点随机抖动；同时整体并发不超过 `maxConcurrency`。**

类比：超市排队结账。

- `minIntervalMs`：收银员处理完一个顾客，至少歇 1.5 秒。
- `jitterMs`：歇的时间在 1.5 ~ 2.5 秒之间随机（避免大家都掐着点同时来）。
- `maxConcurrency`：开几条结账通道。

## 实现：令牌桶 + 队列

[src/core/rate-limiter.ts](../../src/core/rate-limiter.ts)。`createRateLimiter(opts)` 返回一个 `RateLimiter` 对象，只有一个方法：

```typescript
const release = await rateLimiter.acquire(siteKey);
try {
  // 干活 —— 发请求 / 抓数据
} finally {
  release();  // 必须释放
}
```

内部状态：

```typescript
interface PerSiteState {
  inFlight: number;          // 当前正在干活的数量
  lastReleaseAt: number;     // 最近一次 release 的时间戳
  queue: Array<() => void>;  // 在等的队列
}
```

每个 `siteKey` 独立计数——爬 school-bbs 时不影响其他 site。

### `acquire` 流程

1. 把"等待者"塞进 `queue`。
2. 触发 `tryStart(siteKey)`：
   - 看 `inFlight < maxConcurrency` 吗？不行就等。
   - 计算 `waitNeeded = max(0, lastReleaseAt + minIntervalMs - now)`。
   - 加上 `jitter = random() * jitterMs`。
   - `setTimeout` 时间一到，从 `queue` 取一个出来执行，并把 `inFlight++`。
3. 返回一个 `release()` 闭包。调用方干完活调它，`inFlight--` + `lastReleaseAt = now` + `tryStart` 看后面排队的。

### 为什么要 jitter

避免"羊群效应"：

- 没 jitter：多个任务同时排队，都等 `lastReleaseAt + 1500ms`，时间一到全员同时打过去，瞬间高峰。
- 有 jitter：每个延迟随机散开，请求自然均匀分布。

## 配置

来自 `.env`，由 `parseConfig` 读：

| 变量 | 默认 | 含义 |
|---|---|---|
| `RATE_MIN_INTERVAL_MS` | 1500 | 两次请求最小间隔 |
| `RATE_JITTER_MS` | 1000 | 抖动上限（实际间隔在 [min, min+jitter] 之间） |
| `RATE_MAX_CONCURRENCY` | 1 | 同时进行中的请求数（**默认 1 = 串行**） |

**默认串行**——配合保守的 1.5 秒间隔，最常用的"小心运行"模式。开发/测试用即可。

需要快的时候（例如批量初始化）由更上层显式拨快：

- `init:pinned` 通过 yml 里的 `crawl.concurrency`（默认 16）开**多个并行 worker**，每个 worker 各自走限速器。

## 任务里的限速参数

YAML 里有一组更细的间隔，不走 rate-limiter，由 adapter 内部 `page.waitForTimeout` 直接控制：

```yaml
crawl:
  requestIntervalMs: 300              # 帖子之间
  structureRequestIntervalMs: 1500    # 结构页之间（更慢）
  pageTurnIntervalMs: 100             # 翻页之间
```

**和 rate-limiter 的关系**：

- rate-limiter 管"两个**逻辑任务**之间"——例如两个 `CrawlerService.fetchThread()` 调用之间。
- `requestIntervalMs` 等管"一个任务**内部**多次操作之间"——例如一个 `getThread()` 要翻 5 页，每页之间停 100ms。

两层并存：上层粗粒度（不冲整体配额），下层细粒度（不冲单帖速率）。

## 调用约定

`CrawlerService.run()` 包揽了这件事——业务代码**不直接**调 `rateLimiter.acquire`：

```typescript
// crawler-service.ts:run
const release = await this.deps.rateLimiter.acquire(siteKey);
const acquired = await this.deps.browserPool.acquire(siteKey);
try {
  // ... 干活
} finally {
  await page.close();
  acquired.release();
  release();         // ← 这里
}
```

所有 `fetchThread` / `listThreads` / `search` 都经过 `run`，自动受限。

## 出了什么问题

| 现象 | 大概原因 |
|---|---|
| 论坛返回 429 / 提示"操作太频繁" | `minIntervalMs` 太小 / `maxConcurrency` 太大 |
| 爬得非常慢 | 上面两个反过来 / `init:pinned` 没起多线程 |
| 程序卡死不动 | 某个调用方忘了 `release()`（看 `inFlight` 不降说明有泄漏） |

第三种最难排——所以 `release()` 永远写在 `finally` 里，**任何路径都必须释放**。
