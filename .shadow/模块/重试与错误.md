# 模块：重试与错误

> 爬虫面对的是**别人控制的网页**——网络抖、网页慢、被踢出登录、被风控限速……都是常态。所以重试不是优化，是**正常运行的一部分**。错误分门别类，每类配不同的对策。

## 错误分类

[src/core/errors.ts](../../src/core/errors.ts) 定义了一组继承 `BaseAppError` 的具体错误类。每个都有一个稳定的 `code` 字符串：

| `code` | 类 | 什么时候出现 | 怎么处理 |
|---|---|---|---|
| `MISSING_CREDENTIALS` | MissingCredentialsError | env 没设、加密文件也没有 | 不重试，提示用户运行 `npm run login` |
| `LOGIN_FAILED` | LoginFailedError | 账号密码错 / 验证码 / 论坛拒绝 | 不重试，提示 |
| `SESSION_EXPIRED` | SessionExpiredError | 登录态过期，被论坛踢出 | 自动清 storageState + 重登 + 重试 1 次 |
| `NAVIGATION_TIMEOUT` | NavigationTimeoutError | 页面 30s 没加载完 | 自动退避重试 2 次（500ms / 1500ms） |
| `RATE_LIMITED` | RateLimitedError | 论坛主动限速（429 / 反爬提示） | 等 30 秒重试 1 次 |
| `SELECTOR_MISSING` | SelectorMissingError | YAML 里的 CSS 没匹配到 | 不重试，配置错或论坛改版 |
| `UNKNOWN_SITE` | UnknownSiteError | 调了没注册的 siteKey | 不重试，编程错 |
| `BOARD_NOT_FOUND` | BoardNotFoundError | 用户给的版面名字数据库里没有 | 不重试，建议用户先 init / 看名字 |
| `FETCH_FAILED` | FetchFailedError | 其它抓取失败（兜底） | 一般不重试 |
| `DATABASE` | DatabaseError | SQLite 写入失败 / SQL 错 | 不重试（一般是 schema 错或磁盘问题） |

**为什么有 `code`**：让上层（包括将来的 MCP 项目）能稳定判别错误类型，不靠 `instanceof` 或字符串 message 匹配。日志里也直接记 `errorCode`。

## 三层重试

### 第 1 层：`retry()` 工具函数

[src/util/retry.ts](../../src/util/retry.ts)，最朴素的指数退避：

```typescript
await retry(
  () => somethingFlaky(),
  { attempts: 3, baseMs: 1000, factor: 3 },
);
// 尝试 1：立即
// 尝试 2：失败后等 1000ms
// 尝试 3：失败后等 3000ms（1000 * 3）
// 还失败 → 抛最后一次的错
```

可选 `shouldRetry(err)` 判断要不要重试这种错（例如 `LOGIN_FAILED` 就不该重）。

**给谁用**：scripts 和一次性脚本里。例如 `init:pinned` 多线程版本里，每个 worker 调用 `retry()` 包住单帖抓取。

### 第 2 层：`CrawlerService.run()` 的双层重试

[src/core/crawler-service.ts:run](../../src/core/crawler-service.ts)。所有 MCP 风格的工具调用都走这层。

```
外层（最多 2 次）：处理 SessionExpiredError
  │
  └─ 内层（按错误类型）：
       ├─ NavigationTimeoutError → 退避 [500ms, 1500ms]，重试 2 次
       └─ RateLimitedError       → 等 30 秒，重试 1 次
```

**外层**：

- 第 1 次失败如果是 `SessionExpiredError`：清掉 storageState，下一轮重新登录。
- 第 2 次还失败 → 抛出。

**内层**（在外层的每一次尝试里）：

- 单页超时 → 退避重试两次。
- 被限速 → 等 30 秒再来。
- 其它错（DatabaseError、BoardNotFoundError 等）直接抛，不重试。

**两层叠加**最大可能重试次数：2 (outer) × 3 (nav) × 2 (rate) ≈ 极端情况下十几次，但实际上一次会话过期 + 几次网络抖动就解决。

### 第 3 层：`init:pinned` 的多轮重试

[scripts/init/init-pinned.ts](../../scripts/init/init-pinned.ts) 的设计：

- 第 1 轮：用 `crawl.concurrency`（默认 16）并发跑所有版面。
- 失败的版面收集起来。
- 第 2 轮：**并发降为 1** 重跑失败列表。
- 第 3 轮：再来一次（默认 `crawl.maxRetryPasses = 3`）。

理由：高并发时，失败往往是"被论坛瞬间限速"——降并发往往能成。

## 错误怎么往上传

约定：

1. **底层代码（adapter / DOM 解析）**：抛 `BaseAppError` 子类，不要抛 raw `Error`。
2. **repository 层**：所有 `try/catch` 转 `DatabaseError(message, cause)`。
3. **CrawlerService 层**：保留底层抛的错误类型，自己只**包装**`runBodyWithInnerRetries`/`run` 的重试逻辑。
4. **顶层调用方**（scripts、未来的 service）：捕获并打日志，按 `code` 决定退出码 / 用户提示。

不允许：

- `throw new Error("login failed")`（字符串错误）—— 用 `LoginFailedError`。
- 在底层 `console.error` 后 swallow 不抛 —— 错误会丢，上层无感。
- 把原始 error 直接抛上去 —— 失去上下文，上层只看到 `Error: ENOENT: ...` 摸不着头脑。

## 错误码 vs 错误消息

```typescript
catch (e) {
  if (e instanceof BoardNotFoundError) {
    // 用 code 做控制流分支
  }
  const code = (e as { code?: string }).code ?? 'INTERNAL';
  logger.error({ err: String(e), code }, 'task failed');
}
```

- **控制流**用 `instanceof` 或 `e.code`。
- **打日志**记 `code` 字段，便于后续 grep。
- **给人看**用 `e.message`（已经写得人话）。

## 排错小手册

| 错误日志里看到 | 我该看哪 |
|---|---|
| `SESSION_EXPIRED` 反复出现 | storageState 文件有没有写盘成功？凭据正确吗？(`npm run login:once`) |
| `NAVIGATION_TIMEOUT` 频繁 | 网络？换 baseUrl 试试？headless=false 看页面到底卡哪 |
| `RATE_LIMITED` 不断 | 把 `RATE_MIN_INTERVAL_MS` 调大，把 `concurrency` 调小 |
| `SELECTOR_MISSING` | 论坛改版，去 `config/sites/<siteKey>.yml` 里改对应 selector |
| `DATABASE` | `npm run db:check` 看完整性；最近有没有改 schema |
| `BOARD_NOT_FOUND` | `select name from boards;` 看准确名字（注意全角/简繁差异） |
