# 模块：配置加载

> 把 `config/sites/*.yml` 和 `.env` 读进来、检查格式对不对、转成 TypeScript 对象。**所有运行时配置的入口都在这里。**

> 这是"实现层"文档，描述代码怎么写。设计层面"为什么要外置配置"看 [配置文件.md](../配置文件.md)。

## 两个 loader

### `parseConfig(env)` — 读 `.env`

[src/core/config.ts](../../src/core/config.ts) 里。

- 输入：`process.env`（或测试时的伪对象）。
- 输出：`AppConfig`——所有运行时通用参数：数据库目录、浏览器选项、限速参数、日志级别。
- 用 zod schema 校验：

```typescript
const ConfigSchema = z.object({
  DATABASE_PATH:    z.string().default('./.data'),
  BROWSER_HEADLESS: boolFromEnv.default(true),
  STORAGE_STATE_DIR: z.string().default('./.state'),
  IDLE_TIMEOUT_MS:  intFromEnv.default(300_000),
  RATE_MIN_INTERVAL_MS: intFromEnv.default(1_500),
  RATE_JITTER_MS:   intFromEnv.default(1_000),
  RATE_MAX_CONCURRENCY: intFromEnv.default(1),
  LOG_LEVEL: z.enum(['debug', 'info', 'warn', 'error']).default('info'),
  // ...
});
```

`boolFromEnv` 和 `intFromEnv` 是自定义 preprocessor——env 都是字符串，需要先转类型再校验。

**默认值都给齐**：没 `.env` 也能跑。`.env` 主要塞凭据 + 自定义路径。

### `loadSiteConfig(siteKey)` — 读 `config/sites/<siteKey>.yml`

[src/core/site-config.ts](../../src/core/site-config.ts) 里。

- 输入：`siteKey`，例如 `"school-bbs"`。
- 输出：`SiteConfig`——具体某站点的配置：routes / selectors / crawl 参数。
- 用 zod schema 校验（结构复杂，分嵌套 schema：`LoginSelectorsSchema` / `SectionSelectorsSchema` / 等）。
- **内存缓存**：第一次读完缓存在 `Map<siteKey, SiteConfig>`，后续直接命中。重启进程才会重读。
- **siteKey 一致性检查**：YAML 里写的 `siteKey:` 字段必须和文件名一致，否则抛错。这是为了防止"复制粘贴改名字时漏改"。

加载失败的错误信息很详细：

```
Invalid site config config/sites/school-bbs.yml:
  selectors.login.passwordInput: Required
  crawl.boardPages: Expected number, received string
```

### `buildRouteUrl(baseUrl, template, params)`

辅助函数。

```typescript
buildRouteUrl(
  "https://bbs.example.com",
  "/#!board/{key}",
  { key: "BYRatSH" },
);
// → "https://bbs.example.com/#!board/BYRatSH"
```

- `{xxx}` 占位符按 `params.xxx` 替换。
- 占位符在 params 里没值 → 抛错（防止生成像 `/#!board/{key}` 这种带未替换占位符的 URL）。
- `baseUrl` 末尾 `/` 会被去掉，避免双斜杠。

## 凭据 loader（关联但不同模块）

[`getCredentials(siteKey, env)`](../../src/core/config.ts) 不属于"配置"语义，但实现上也在 config.ts 里：

- 按 siteKey 大写化拼出环境变量名：例如 `siteKey: 'school-bbs'` → `SCHOOL_BBS_USERNAME` / `SCHOOL_BBS_PASSWORD` / `SCHOOL_BBS_BASE_URL` / `SCHOOL_BBS_LOGIN_URL`。
- 缺 username 或 password → 抛 `MissingCredentialsError`，错误对象里带 `missingEnvKeys`。
- `baseUrl` 和 `loginUrl` 可选。

详见 [浏览器与会话.md](浏览器与会话.md) §"凭据获取"。

## zod 速记

写一遍配置 schema 的常用片段：

```typescript
z.string()                            // 字符串
z.string().min(1)                     // 非空字符串
z.number().int().positive()           // 正整数
z.number().int().nonnegative()        // 非负整数
z.enum(['a', 'b', 'c'])               // 枚举
z.object({ name: z.string() })        // 对象
z.array(z.string())                   // 字符串数组
z.preprocess(v => ..., z.boolean())   // 用法：把环境变量字符串"true"转 boolean

const Result = MySchema.safeParse(data);
if (!Result.success) { console.log(Result.error.issues); }
```

**为什么用 zod 而不是 JSON Schema**：

- zod 类型即 schema——`z.infer<typeof X>` 直接拿 TS 类型。
- 错误信息友好（带路径），不用配 JSON Schema 的额外工具链。
- 体积小。
