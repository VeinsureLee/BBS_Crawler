# 配置文件

> 论坛的样子（哪些讨论区、网页长啥样、按钮在哪）是会变的。代码里如果写死，论坛改版一次代码就改一次。所以这些"会变的东西"全部抽到 **YAML 配置文件**里，代码只读不写、运行时按配置工作。

## 一句话原则

**代码里不允许出现具体的讨论区名、版面 key、CSS 选择器、URL 路径模板。这些都从 `config/` 下的 YAML 读。**

## 三类配置文件

```
config/sites/
  school-bbs.yml             ← 网页结构 + 爬取参数（已存在）
  school-bbs.entries.yml     ← 论坛入口清单（待新增）
  school-bbs.node-types.yml  ← 节点类型描述（待新增）

.env                          ← 密码、baseUrl 等机密（不进 git）
```

下面分别说。

### A. `<siteKey>.yml` — 网页结构 + 爬取参数

**作用**：告诉爬虫"登录框在哪、版面表格的行长啥样、每次请求间隔多久"等技术细节。

**示例**（实际是 `config/sites/school-bbs.yml`）：

```yaml
siteKey: school-bbs
displayName: School BBS

routes:
  section: "/#!section/{key}"           # 讨论区页面 URL 模板
  board:   "/#!board/{key}"
  thread:  "/#!article/{boardKey}/{threadId}"

selectors:
  login:
    form: '#f_login'                    # 登录表单
    usernameInput: 'input[name="id"]'
    passwordInput: 'input[name="passwd"]'
    submitButton: 'input[type="submit"]'
    loggedInIndicator: '#u_login'       # 出现这个元素 = 已登录
  section:
    boardRowReady: 'td.title_1'         # 等这个出现，说明版面列表加载好了
    sectionLinks: '#xlist a[href^="/section/"]'
  board:
    threadRowReady: 'a[href^="/article/"]'

crawl:
  boardPages: 3                         # 每个版面默认爬几页
  concurrency: 16                       # init:pinned 的并发度
  requestIntervalMs: 300                # 帖子之间间隔
  structureRequestIntervalMs: 1500      # 结构页之间间隔（更慢）
  pageTurnIntervalMs: 100               # 翻页之间间隔
  maxPinnedThreadPages: 10              # 单个置顶帖最多爬几页
  maxRetryPasses: 3                     # 失败重试轮次
```

**为什么这样**：

- `routes` 是 URL 模板。`{key}` 占位符运行时替换。改版（例如新版换成 `/forum/section/...`）只改一行。
- `selectors` 全是 CSS。论坛改 DOM 类名，只改这里。
- `crawl.*` 是性能参数。觉得太慢/太激进，改数字就行，不动代码。

### B. `<siteKey>.entries.yml` — 论坛入口清单

**已落地**（Phase 2）。当 `entries.yml` 存在且 `forums:` 非空时，`init-sections` **直接读这个文件**，跳过爬首页。缺失或空列表则回退到 adapter.listSections（打 warn 日志）。

**示例**：

```yaml
siteKey: school-bbs

# 顶级讨论区清单（手工维护，半年看一次有没有新增）
forums:
  - sectionKey: ten            # 内部 key（来自论坛 URL 的 /section/ten）
    name: 本站十大              # 人类可读名
    nodeType: forum            # 可选，引用 node-types.yml；缺省 "forum"
  - sectionKey: school
    name: 校园生活
  - sectionKey: club
    name: 社团
  # ... 大约 10 个
```

**字段规则**：

- `sectionKey`：必填，必须唯一，对应论坛 URL 里的 `/section/{key}`
- `name`：必填，给人看的中文名
- `nodeType`：可选，默认 `"forum"`；若提供必须在 [node-types.yml](#c-sitekeynode-typesyml--节点类型描述) 里声明

**校验**：加载时 zod 验证字段、检测重复 `sectionKey`；启动时 `validateConfigConsistency` 会确保 `nodeType` 引用合法。

**好处**：

- 想短期只爬"社团"测试？注释掉别的，跑一遍。
- 论坛新加一个"研究生专区"？加一行配置，不改代码。
- 别人 fork 项目爬别的学校论坛？只改这个文件 + selectors。

**首次填充工作流**：
1. 第一次跑 `npm run init:sections` 让 adapter 从首页发现讨论区（fallback 路径），落到数据库。
2. 把数据库里的 `sections` 表手工抄进 `entries.yml`。
3. 之后所有 init 都走配置驱动；首页改版不再影响 init。

### C. `<siteKey>.node-types.yml` — 节点类型描述

**已落地**（Phase 2）。本文件**只做声明 + 校验**，不直接驱动解析逻辑——解析仍在 adapter 代码里。Phase 3 数据库分层落地后，可能会用 `nodeType` 字段决定一个节点存到哪个 `.db` 文件。

**作用**：论坛节点虽然名字不同，但**形态**就那几种，分层数据库 schema 也用同一个枚举（`forum` / `sub_forum` / `board` / `thread`）。把它们集中声明一次，让 entries.yml 引用。

**示例**（即 `config/sites/school-bbs.node-types.yml`）：

```yaml
siteKey: school-bbs

nodeTypes:
  forum:
    description: 顶级讨论区（数据库 type='forum'，每个对应一个独立 .db 文件）
    childTypes: [sub_forum, board]

  sub_forum:
    description: 二级 / 三级讨论区，本身不存帖，只是组织结构
    childTypes: [board, sub_forum]

  board:
    description: 版面（叶子节点的容器），底下是帖子
    childTypes: [thread]

  thread:
    description: 帖子，最末梢，包含若干楼层
    childTypes: []
```

**字段规则**：

- `nodeTypes` 是个 map，key 是类型名（自取）
- `description`：必填，给人看
- `childTypes`：可选，默认空数组；每个值必须是同文件里声明过的类型名（loader 校验）

**为什么不放 `parser` 字段**：早期设计想过把 parser 函数名也外置，但"如何从 DOM 抽数据"本质上是代码——配置不能消除解析逻辑，只能消除"叫谁、长几层"的元数据。所以 parser 仍在 adapter 里硬写。

## 加载流程

1. Node 启动时，`src/core/config.ts` 读 `.env`，拿到 `SCHOOL_BBS_BASE_URL` / `SCHOOL_BBS_USERNAME` 等。
2. 第一次用到某个 site 时，三个 loader 各自按需读对应文件、zod 校验、内存缓存：
   - `loadSiteConfig(siteKey)` ← `<siteKey>.yml`（selectors + routes + crawl）
   - `loadSiteEntries(siteKey)` ← `<siteKey>.entries.yml`（顶级讨论区清单）
   - `loadNodeTypes(siteKey)` ← `<siteKey>.node-types.yml`（节点形态）
3. 后两个 loader **文件缺失时返回 null**——调用方决定走配置驱动还是 fallback。
4. `validateConfigConsistency(siteKey)` 跨文件校验：entries 里的 `nodeType` 必须在 node-types 里存在。在 init 启动时调一次。
5. 任何校验失败直接抛错，**不让一个错配置进入运行时**。
6. **测试隔离**：loader 读 `process.env.SITE_CONFIG_DIR`（默认 `./config/sites/`），测试用 tmp 目录覆盖。

## 怎么加一个新论坛

假设要支持 `xx-university-bbs`：

1. `.env` 加 4 个环境变量：`XX_UNIVERSITY_BBS_BASE_URL` / `_USERNAME` / `_PASSWORD` / `_LOGIN_URL`。
2. 新建 `config/sites/xx-university-bbs.yml`：抄一份 school-bbs.yml，改 selectors / routes。
3. 新建 `config/sites/xx-university-bbs.entries.yml`：列出讨论区清单。
4. 新建 `config/sites/xx-university-bbs.node-types.yml`：描述节点形态。
5. 新建 `src/adapters/xx-university-bbs/`：实现 `SiteAdapter` 接口，引用上面的 parser 名字。在 `src/adapters/index.ts` 加一行 `import './xx-university-bbs';`。
6. 跑 `npm run init -- --site xx-university-bbs`。

理论上**步骤 1-4 可以由非开发者完成**，步骤 5 不行（需要写代码）。但 adapter 那部分代码很薄，主要是几个 DOM 解析函数。

## 凭据与配置分离

**绝对不要**把密码写进 `config/sites/*.yml`——这些文件**会进 git**，公开。

机密走两条路径：

1. `.env` 文件（开发机本地）。`.env` 在 `.gitignore` 里。
2. 加密凭据存储 `./.state/<siteKey>.credentials.enc`。由 `npm run login` 命令交互式录入并加密保存（AES-256-GCM，密钥从主机名派生）。详见 [模块/浏览器与会话.md](模块/浏览器与会话.md)。
