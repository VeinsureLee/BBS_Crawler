# 数据库

> 数据库就是"放数据的柜子"。本项目的柜子被故意拆成很多小柜子，按"讨论区"分。这样查一个讨论区时不用动别的，搬整个数据库时也只搬需要的那几格。

## 一句话设计

**一个讨论区一个 SQLite 文件（`.db`）；顶层用一张"索引表"记住每个文件在哪。**

类比一下：

- 总目录：图书馆门口的指示牌（一张小表，标注"文学区在 2 楼，科技区在 3 楼"）
- 各分馆：每层楼的具体书架（独立运转，互不干扰）

不是把所有书都堆一个房间，而是用"指针/路径"指过去。这是 C 语言里指针的思路，把"数据本体"和"数据位置"分开。

## 为什么这样设计

| 痛点 | 单库做法 | 分层做法 |
|---|---|---|
| 备份 1 个讨论区 | 全库 dump | 拷一个 `.db` 文件 |
| 删除/重置 1 个讨论区 | DELETE + VACUUM | `rm` 文件 |
| 并发写不同讨论区 | 互相锁 | 文件互不相关 |
| 一个文件损坏 | 全完 | 只丢一个讨论区 |
| 排查单个讨论区数据 | 在大表里筛 | 直接打开那一个 `.db` |

代价是**跨讨论区查询**会麻烦点（需要打开多个 `.db`），但本项目几乎没有这种需求——上层的整理、智能检索由**外部 MCP 项目**做，那边可以自己决定要不要做物化视图。

## 物理布局

```
<DATABASE_PATH>/                    ← 来自 .env 的 DATABASE_PATH（默认 ./.data）
  structure.db                      ← 索引层：论坛骨架 + 指针 + 站点元 + 调用审计
  forums/
    本站十大.db                       ← 内容层：每个顶级讨论区一个文件
    社团.db
    校园生活.db
    ...
```

文件名取讨论区 sectionKey 的"安全版"（非字母数字字符替换成 `_`），具体路径由 `structure.db.nodes.db_file` 字段记下——**不是靠路径约定靠记忆**。

## 索引层：`structure.db`

三张表：`sites` / `nodes` / `fetch_log`（外加 `migrations` 用于未来 schema 演进的占位）。

### `nodes`：递归式骨架

整个论坛的层级（讨论区 → 子讨论区 → 版面）**全部存在一张表里**，通过 `parent_id` 自指。实测最深 3 层（顶级 → 子讨论区 → 版面），但表本身支持任意深度。

```sql
CREATE TABLE nodes (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  parent_id       INTEGER REFERENCES nodes(id) ON DELETE CASCADE,
  site_key        TEXT NOT NULL REFERENCES sites(site_key) ON DELETE CASCADE,
  node_key        TEXT NOT NULL,                       -- 论坛 URL 里的 sectionKey / boardKey
  name            TEXT NOT NULL,                       -- 给人看的名字
  type            TEXT NOT NULL CHECK (type IN ('forum','sub_forum','board')),
  level           INTEGER NOT NULL,                    -- 0 = 顶级讨论区
  db_file         TEXT,                                -- ← 指针：相对 dataDir 的路径，**仅 forum 类型填写**
  moderators      TEXT,                                -- JSON, 仅 board 类型
  stats           TEXT,                                -- JSON, 仅 board 类型
  raw             TEXT,                                -- 任意原始数据
  created_at      TEXT NOT NULL DEFAULT (datetime('now')),
  last_crawled_at TEXT,
  UNIQUE (site_key, node_key)
);

CREATE INDEX idx_nodes_parent     ON nodes(parent_id);
CREATE INDEX idx_nodes_site_type  ON nodes(site_key, type);
```

字段说明：

- **`parent_id`**：上一级讨论区的 `id`。顶级讨论区为 `NULL`。
- **`type`**：节点是论坛/子论坛/版面之一。`board` 表示叶子——版面下面是帖子，不是子讨论区。
- **`db_file`**：**只有 `type='forum'` 的顶级讨论区填**，例如 `"forums/club.db"`。下层节点（sub_forum、board）的实际帖子数据全部存进**它所属顶级讨论区的那个 `.db`**。
- **`moderators` / `stats`**：仅 `type='board'` 时有意义。当作 JSON 字符串存。
- **`UNIQUE(site_key, node_key)`**：同一个站点下，node_key（即 URL 里的那一段）必须唯一。

### `sites` 和 `fetch_log`

```sql
CREATE TABLE sites (
  site_key     TEXT PRIMARY KEY,
  display_name TEXT NOT NULL,
  base_url     TEXT NOT NULL,
  created_at   TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE fetch_log (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  site_key    TEXT NOT NULL,
  tool        TEXT NOT NULL,
  args        TEXT NOT NULL,
  status      TEXT NOT NULL,
  error_code  TEXT,
  duration_ms INTEGER,
  created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

`fetch_log` 从老 `content.db` 搬到了这里——它是**全局调用审计**，跟具体讨论区没关系，放在索引层更合适。

### 查"某个版面属于哪个 .db 文件"

递归 CTE 往上走到 `type='forum'` 那一行，拿它的 `db_file`：

```sql
WITH RECURSIVE up(node_id) AS (
  SELECT $1            -- starting board's node id
  UNION ALL
  SELECT n.parent_id FROM up u JOIN nodes n ON n.id = u.node_id
   WHERE n.parent_id IS NOT NULL
)
SELECT n.db_file FROM up u JOIN nodes n ON n.id = u.node_id
 WHERE n.type = 'forum' LIMIT 1;
```

实现见 [src/repository/boards.ts:findForumDbFileForBoard](../src/repository/boards.ts)。

## 内容层：每个讨论区一个 `.db`

每个顶级讨论区的 `.db` 文件里固定 **4 张数据表**（外加 `migrations` 占位）：

```sql
-- 1. 帖子元数据（pinned/regular 混存，用 is_pinned 区分）
CREATE TABLE threads (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  board_node_id   INTEGER NOT NULL,                  -- 引用 structure.db.nodes.id（跨库，无强制 FK）
  url             TEXT NOT NULL,
  title           TEXT NOT NULL,
  author          TEXT,
  posted_at       TEXT,                              -- ISO 8601, 排序键
  last_reply_at   TEXT,
  reply_count     INTEGER,
  view_count      INTEGER,
  raw             TEXT,
  is_pinned       INTEGER NOT NULL DEFAULT 0,
  first_seen_at   TEXT NOT NULL DEFAULT (datetime('now')),
  last_fetched_at TEXT NOT NULL DEFAULT (datetime('now')),
  UNIQUE (url)
);

CREATE INDEX idx_threads_board         ON threads(board_node_id, posted_at DESC);
CREATE INDEX idx_threads_board_pinned  ON threads(board_node_id, is_pinned, posted_at DESC);

-- 2. 楼层内容
CREATE TABLE posts (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  thread_id     INTEGER NOT NULL REFERENCES threads(id) ON DELETE CASCADE,
  floor         INTEGER NOT NULL,
  author        TEXT NOT NULL,
  posted_at     TEXT,
  content_html  TEXT NOT NULL,
  content_text  TEXT NOT NULL,
  attachments   TEXT,
  raw           TEXT,
  UNIQUE (thread_id, floor)
);

-- 3. 每版面的爬取进度（watermark + 最深页）
CREATE TABLE board_crawl_state (
  board_node_id           INTEGER PRIMARY KEY,
  deepest_page_crawled    INTEGER NOT NULL DEFAULT 0,
  latest_thread_posted_at TEXT,
  last_crawled_at         TEXT,
  last_thread_key         TEXT
);

-- 4. 每日流量快照（可选，由后续"统计快照"作业灌入；init 不主动填）
CREATE TABLE daily_traffic (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  board_node_id INTEGER NOT NULL,
  date          TEXT NOT NULL,
  online        INTEGER,
  today_posts   INTEGER,
  threads       INTEGER,
  posts         INTEGER,
  UNIQUE (board_node_id, date)
);
```

### 为什么 pinned 不另开表

早期设计想过 `pinned_threads` + `pinned_posts` 独立两张表。最终合并到 `threads`/`posts` 共用，理由：
- 现在的写入路径有 `is_pinned = is_pinned OR $9` 的 OR-merge 语义，**pinned 标志一旦置上不会被后续 upsert 抹掉**（参见 [src/repository/threads.ts](../src/repository/threads.ts)）。
- 分两套表会让"普通帖升级为置顶"或反向降级需要跨表迁移，复杂度上升。
- 查询模式（"某版最新 N 帖"或"某版置顶"）通过 `(board_node_id, is_pinned, posted_at)` 复合索引足以快速分流。

### 没有 `boards` 表

老 schema 有独立的 `boards` 表（id / board_key / name / moderators / stats / section_id）。Phase 3 把这些信息全部并入 `nodes` 表（`type='board'`），不再单列。理由：
- 一个 board 在系统里就是一个节点，有 id / 父节点 / 元数据 —— 全部可以放在 `nodes` 一行里。
- 内容（threads + posts）才是分层时需要按讨论区分开的部分。结构性元数据放在索引库里更合理。
- 少一张表 = 少一处需要同步的 FK = 迁移和查询都更简单。

`moderators` / `stats` 作为 JSON 列存在 `nodes` 行上，按 `type='board'` 才有意义。

## 连接管理

`src/repository/db.ts` 提供：

- `initDb({ dataDir })`：打开 `<dataDir>/structure.db`，应用 schema（CREATE TABLE IF NOT EXISTS，幂等）
- `getStructureDb()`：取索引库单例
- `getForumDb(dbFile)`：按相对路径打开/缓存一个 forum db，**第一次打开时自动应用 schema**——所以"新建讨论区"无需任何 DDL 步骤，只需调一次 `getForumDb` 即可
- `closeAllDbs()`：关掉所有 db 句柄（关浏览器进程退出前调）

forum db 的连接缓存在 `Map<dbFile, SQLiteDb>` 里，进程生命周期内一直保持。无 LRU 淘汰——总数最多十来个，不会爆 fd。

## 迁移路径

老布局（Phase 3 之前）：
- `structure.db`: sites / sections / boards / board_crawl_state
- `content.db`: threads / posts / fetch_log

新布局（Phase 3 之后，本文档描述的状态）：
- `structure.db`: sites / nodes / fetch_log
- `forums/<key>.db`: threads / posts / board_crawl_state / daily_traffic

**一次性迁移脚本**：[scripts/db/migrate-to-layered.ts](../scripts/db/migrate-to-layered.ts)

```bash
# 试跑（不写文件，只打印计划）
npm run db:migrate:layered -- --dry-run

# 真跑（默认 source=./data, target=./.data）
npm run db:migrate:layered

# 自定义路径
npm run db:migrate:layered -- --source ./old-data --target ./.data --yes
```

脚本流程：
1. 读旧 `structure.db` + `content.db`；
2. 按递归父子关系拓扑排序，生成 `nodes` 行；
3. 旧 `boards` → `nodes` (`type='board'`)，旧 sections 拓扑顺序逐条建 `forum`/`sub_forum`；
4. 按 (site, board) 找到所属 forum，把对应的 threads / posts / board_crawl_state 灌进 `forums/<key>.db`；
5. fetch_log 整体搬到新 structure.db；
6. 写入暂存目录 `<target>.staging-<ts>/`；
7. **校验行数**：sites/nodes/fetch_log 严格相等；threads/posts 允许"漏的少于等于旧的"（孤儿帖子被跳过 + 记 warn 日志）；
8. 成功后：原 `<target>` → `<target>.backup-<ts>/`，暂存 → `<target>`，原子 rename。

失败时暂存目录自动清理，原 `<target>` 不动。
