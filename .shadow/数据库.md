# 数据库

> 数据库就是"放数据的柜子"。本项目的柜子被故意拆成很多小柜子，按"讨论区"分。这样查一个讨论区时不用动别的，搬整个数据库时也只搬需要的那几格。

## 一句话设计

**一个讨论区一个 SQLite 文件（`.db`）；顶层用一张"索引表"记住每个文件在哪。**

类比一下：

- 总目录：图书馆门口的指示牌（一张小表，标注"文学区在 2 楼，科技区在 3 楼"）
- 各分馆：每层楼的具体书架（独立运转，互不干扰）

不是把所有书都堆一个房间，而是用"指针/路径"指过去。这是 C 语言里指针的思路，把"数据本体"和"数据位置"分开。

## 为什么这样设计

| 痛点 | 单库做法 | 分层做法 |
|---|---|---|
| 备份 1 个讨论区 | 全库 dump | 拷一个 `.db` 文件 |
| 删除/重置 1 个讨论区 | DELETE + VACUUM | `rm` 文件 |
| 并发写不同讨论区 | 互相锁 | 文件互不相关 |
| 一个文件损坏 | 全完 | 只丢一个讨论区 |
| 排查单个讨论区数据 | 在大表里筛 | 直接打开那一个 `.db` |

代价是**跨讨论区查询**会麻烦点（需要打开多个 `.db`），但本项目几乎没有这种需求——上层的整理、智能检索由**外部 MCP 项目**做，那边可以自己决定要不要做物化视图。

## 物理布局

```
.data/                          ← 配置项 DATABASE_PATH
  structure.db                  ← 索引层：论坛骨架 + 指针
  forums/
    本站十大.db                  ← 内容层：每个顶级讨论区一个文件
    社团.db
    校园生活.db
    ...
```

文件名取讨论区的可读名字（例如 `本站十大.db`）。具体文件位置由 `structure.db.nodes.db_file` 这个字段记下，**不是靠路径约定靠记忆**。

## 索引层：`structure.db`

只有一张核心表，递归式结构（同一张表里通过 `parent_id` 自指）。论坛的"讨论区 → 二级讨论区 → 三级讨论区"无论几层都能容纳，**实测最深 3 层**。

```sql
CREATE TABLE nodes (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  parent_id   INTEGER REFERENCES nodes(id) ON DELETE CASCADE,
  name        TEXT    NOT NULL,                    -- "本站十大"
  type        TEXT    NOT NULL CHECK (type IN ('forum','sub_forum','board')),
  level       INTEGER NOT NULL,                    -- 0 = 顶级讨论区
  db_file     TEXT,                                -- ← 这就是"指针"，相对 .data/ 的路径
  raw         TEXT,                                -- 原始 JSON，留作回溯
  created_at  INTEGER NOT NULL DEFAULT (unixepoch()),
  UNIQUE (parent_id, name)
);

CREATE INDEX idx_nodes_parent  ON nodes(parent_id);
CREATE INDEX idx_nodes_type    ON nodes(type);
```

字段说明：

- **`parent_id`**：上一级讨论区的 `id`。顶级讨论区为 `NULL`。
- **`type`**：节点是论坛/子论坛/版面之一。`board` 表示叶子——版面下面是帖子，不是子讨论区。
- **`db_file`**：**只有 `type='forum'` 的顶级讨论区填**，例如 `"forums/本站十大.db"`。下层节点（sub_forum、board）的实际帖子数据全部存进**它所属顶级讨论区的那个 `.db`**。
- **`UNIQUE(parent_id, name)`**：同一个父节点下名字不能重复。

### 查"某个版面属于哪个 .db 文件"

递归往上查到顶层，拿那一行的 `db_file`：

```sql
WITH RECURSIVE ancestors(id, parent_id, db_file, level) AS (
  SELECT id, parent_id, db_file, level FROM nodes WHERE id = ?
  UNION ALL
  SELECT n.id, n.parent_id, n.db_file, n.level
    FROM nodes n
    JOIN ancestors a ON n.id = a.parent_id
)
SELECT db_file FROM ancestors WHERE level = 0;
```

## 内容层：每个讨论区一个 `.db`

每个顶级讨论区的 `.db` 文件里固定 **6 张表**：

```sql
-- 1. 版面元数据（这个讨论区下的所有版面，含子讨论区下的）
CREATE TABLE boards (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  node_id      INTEGER NOT NULL,        -- structure.db.nodes.id 的拷贝（跨库引用）
  board_key    TEXT NOT NULL UNIQUE,    -- 论坛内部 key，例如 "BYRatSH"
  name         TEXT NOT NULL,
  moderators   TEXT,                    -- JSON 数组
  stats_snapshot TEXT,                  -- JSON：在线、今日、主题数、回复数 + snapshotAt
  last_crawled_at INTEGER
);

-- 2. 置顶帖元数据（一个版面可以有多个置顶）
CREATE TABLE pinned_threads (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  board_id    INTEGER NOT NULL REFERENCES boards(id) ON DELETE CASCADE,
  thread_key  TEXT NOT NULL,            -- "{boardKey}/{articleId}"
  title       TEXT NOT NULL,
  author      TEXT,
  posted_at   INTEGER,
  last_fetched_at INTEGER,
  UNIQUE (board_id, thread_key)
);

CREATE INDEX idx_pinned_board ON pinned_threads(board_id);

-- 3. 置顶帖内容（楼层）
CREATE TABLE pinned_posts (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  pinned_id     INTEGER NOT NULL REFERENCES pinned_threads(id) ON DELETE CASCADE,
  floor         INTEGER NOT NULL,
  author        TEXT,
  posted_at     INTEGER,
  content_text  TEXT,
  raw           TEXT,
  UNIQUE (pinned_id, floor)
);

-- 4. 普通帖子元数据
CREATE TABLE threads (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  board_id    INTEGER NOT NULL REFERENCES boards(id) ON DELETE CASCADE,
  thread_key  TEXT NOT NULL,
  title       TEXT NOT NULL,
  author      TEXT,
  posted_at   INTEGER,                  -- 排序键（见下）
  last_reply_at INTEGER,
  reply_count INTEGER,
  view_count  INTEGER,
  last_fetched_at INTEGER,
  UNIQUE (board_id, thread_key)
);

CREATE INDEX idx_threads_board_posted ON threads(board_id, posted_at DESC);

-- 5. 普通帖子内容
CREATE TABLE posts (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  thread_id     INTEGER NOT NULL REFERENCES threads(id) ON DELETE CASCADE,
  floor         INTEGER NOT NULL,
  author        TEXT,
  posted_at     INTEGER,
  content_text  TEXT,
  raw           TEXT,
  UNIQUE (thread_id, floor)
);

-- 6. 每日流量快照（可选填，用来观察趋势）
CREATE TABLE daily_traffic (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  board_id    INTEGER NOT NULL REFERENCES boards(id) ON DELETE CASCADE,
  date        TEXT NOT NULL,            -- "YYYY-MM-DD" CST
  online      INTEGER,
  today_posts INTEGER,
  threads     INTEGER,
  posts       INTEGER,
  UNIQUE (board_id, date)
);
```

### 排序约定：用时间戳

按你之前的拍板：**排序键是 `posted_at`**（Unix 秒）。
- 索引 `idx_threads_board_posted` 让"某版最新 100 帖"瞬间出。
- 不做全文索引（FTS）。如果以后下游需要全文检索，由 MCP 项目按需建。

## 连接管理

打开 SQLite 用 LRU 池——最多同时开 N 个文件（N 通过环境变量配，默认 8）。冷文件 60 秒空闲后自动关。

- 跨库**只读 join** 走 `ATTACH DATABASE`（最多 attach 几个就够，几乎用不到）。
- 跨库**写**永远是先写一个，提交后再写另一个，不跨库事务。

## 当前状态 vs 目标状态

**目前代码（`src/repository/db.ts`）还没分层。** 它只开了两个固定的库：

- `structure.db`：sites / sections / boards / board_crawl_state
- `content.db`：threads / posts / fetch_log

本设计是**目标状态**，按你之前对话里的拍板。要落到代码上需要一次性迁移：

1. 写迁移脚本（沿用 `scripts/db/migrate.ts` 框架）：
   - 读旧 `structure.db.sections` → 新 `structure.db.nodes`（按 `parent_section_id` 拉成树）
   - 读旧 `structure.db.boards` → 各顶级讨论区的 `forums/*.db`
   - 读旧 `content.db.threads/posts` → 拆到对应的 `forums/*.db`
   - 校验：行数对齐、抽样比对内容
2. 切换 `src/repository/db.ts`：把 `initDbs`/`getStructureDb`/`getContentDb` 换成 `getStructureDb()` + `getForumDb(forumName)`。
3. `repository/*` 里所有用到 `getContentDb()` 的函数改为先查 `structure.db.nodes` 找到 `db_file`，再打开对应 forum db。

这个迁移是**一次性**的——上线后回不去。先在新分支跑通、对照通过，再 merge 主干。
