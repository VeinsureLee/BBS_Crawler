# 工作流程 1：初始化

> "初始化"就是把论坛的骨架（讨论区→版面→置顶帖）**从零抓一遍**塞进数据库。第一次部署项目时跑一次；之后每隔一段时间（例如一周）跑一次，刷新结构变化。

## 一句话流程

```
登录 → 抓讨论区清单 → 抓每个讨论区的版面 → 抓每个版面的置顶帖
```

三步骤连成一条流水线，可以一键跑（`npm run init`），也可以拆开单跑某一步。

## 三个步骤详解

### 步骤 1：抓讨论区清单（`npm run init:sections`）

**目的**：把论坛有哪些讨论区记录下来，例如"本站十大""校园生活""社团"……

**做法**（**目标状态**——按 [配置文件.md](../配置文件.md) 那边的 B 节）：

- 读 `config/sites/<siteKey>.entries.yml` 里的 `forums:` 列表。
- 对每条入口调用 `upsertSection`/`upsertNode` 写进 `structure.db`（顶级，`parent_id = NULL`）。

**做法（当前状态）**：

- 打开论坛首页，按 `selectors.section.sectionLinks` 这条 CSS 选择器找到所有 `<a href="/section/...">`。
- 抓到名字和 `sectionKey`，逐条 upsert。
- 见 `src/adapters/school-bbs/index.ts:listSections`。

**为什么要从首页抓换成读配置**：参见 [配置文件.md](../配置文件.md) §B。简言之：解耦，且不让论坛首页改版连带整个初始化失败。

### 步骤 2：抓版面结构（`npm run init:boards`）

**目的**：对每个讨论区，把它下面的版面（以及可能存在的二级/三级子讨论区）抓下来。

**做法**：

- 遍历 `structure.db` 里所有顶级讨论区（`level = 0`）。
- 对每个讨论区，访问其页面，按 `selectors.section.boardRowReady` 解析 `<table.board-list>`。
- 一行行解析，区分**版面**（href 形如 `/board/{key}`）和**子讨论区**（额外有"[二级目录]"标记，href 形如 `/section/{key}`）。
- 版面 → `upsertBoard()`，子讨论区 → `upsertSection()` 后**递归一次**进入子讨论区抓里面的版面。
- 一并把版面的运营数据（在线人数、今日发帖、版主列表）作为快照存入 `boards.stats`。

**只递归一层**：根据观察，school-bbs 最深就 3 层（顶级 → 子讨论区 → 版面）。多层结构以后再扩展。

### 步骤 3：抓每个版面的置顶帖（`npm run init:pinned`）

**目的**：把每个版面的置顶帖（最多几个）的**全部楼层内容**抓回来。普通帖子留给工作流程 2 按需抓。

**做法**：

- 遍历 `structure.db` 里所有 `boards`。
- 访问 `/#!board/{boardKey}`，找 `<tr class="top">` 的行——这些就是置顶。提取每条的 `articleId`。
- 对每个 `articleId` 拼出帖子 URL `/article/{boardKey}/{articleId}`，调用 adapter 的 `getThread()`。
- `getThread()` 会自动翻页（最多 `crawl.maxPinnedThreadPages` 页，防止超长帖跑飞），收集所有楼层。
- 通过 `upsertThread({ isPinned: true })` + `upsertPosts()` 落库。

**并发**：这一步会发起很多 HTTP 请求，所以**唯独这一步开了多线程**——通过 Playwright 多个 page/context 并行，并发度由 `crawl.concurrency`（默认 16）控制。注意限速器 [模块/限速.md](../模块/限速.md) 会兜底，单 siteKey 全局有最小间隔。

## 一键跑 vs 分步跑

`package.json` 里：

```json
"init:sections": "tsx scripts/init/init-sections.ts",
"init:boards":   "tsx scripts/init/init-boards.ts",
"init:pinned":   "tsx scripts/init/init-pinned.ts",
"init":          "npm run init:sections && npm run init:boards && npm run init:pinned"
```

- **首次部署**：直接 `npm run init`。
- **只想刷新版面列表**（例如知道有几个新版面）：`npm run init:boards`。
- **某次跑挂了断在中间**：从下一步继续就行，每一步都是 upsert + 幂等。

## 智能续跑机制

`src/core/init-orchestrator.ts` 在每次有外部调用进来时**主动检查**数据库状态：

| 状态 | 行为 |
|---|---|
| `sections` 表为空 | 全量 init（三步都跑） |
| 某些 section 没有 board | 只补这些 section 的 step 2 + step 3 |
| 某些 board 没有置顶帖 | 只补这些 board 的 step 3 |
| 全有 | 跳过（缓存命中后即使再调也立即返回） |

这样**第一次使用项目的 agent 不用手动跑 init**——它会触发一次惰性初始化，之后命中缓存。

## 多线程与重试

按你之前明确的要求，**初始化时的多线程重试机制保持现状**：

### 多线程

只在 `init:pinned` 步骤启用。`scripts/init/init-pinned.ts` 用 worker pool 模式：

- 主线程从队列里发版面给 worker。
- 每个 worker 用独立的 BrowserContext（不同的 storageState/cookie 复用）。
- 单 siteKey 的限速器（[模块/限速.md](../模块/限速.md)）保证总速率不会过载。

### 重试

参见 [模块/重试与错误.md](../模块/重试与错误.md)。归纳：

1. **`retry()` 工具函数**：单个网络请求/页面操作出错 → 指数退避重试，默认 3 次。
2. **重试轮次**：`crawl.maxRetryPasses`（默认 3）。一轮跑完后，把所有失败的版面集中起来，再用**并发度 1**重跑一次，然后是 2 次、3 次……减小并发避开过载。
3. **会话过期**：检测到 `SessionExpiredError` → 自动重登（清掉 storageState 重走 login）。

## 失败/中断怎么办

| 现象 | 应对 |
|---|---|
| 网络中断，某步骤挂了 | 直接重跑这一步，已 upsert 的不会重抓 |
| 论坛改 DOM，某个 selector 失效 | 改 `config/sites/<siteKey>.yml` 里对应 selector，重跑 |
| 某些版面持续失败（例如权限受限） | 进 `npm run debug:failed-boards` 看清单，决定跳过还是修 |
| 数据库脏了，想推倒重来 | `npm run db:delete-pinned`（只删置顶）或备份后 `rm .data/*.db` 全删 |

## 命令行入口一览

```bash
# 第一次或全量刷新
npm run init                  # 串行跑三步

# 单步
npm run init:sections         # 仅讨论区清单
npm run init:boards           # 仅版面结构
npm run init:pinned           # 仅置顶帖

# 辅助
npm run init:export           # 把骨架导出成 JSON（不含帖子内容）
npm run db:check              # 检查数据库完整性
npm run debug:failed-boards   # 查看上次哪些版面没抓成功
```
